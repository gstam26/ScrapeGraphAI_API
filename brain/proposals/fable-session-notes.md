# Fable session notes — lessons log

One entry per lesson; one-line summary on top of each. Both corrections and confirmed approaches. Newest first.

## A populated cache doesn't freeze a pipeline stage unless ALL of that stage's inputs are persisted — and check the baseline's provenance before promising a replay
I advised "don't clear the cache — the re-run replays the 07-03 baseline identically, zero credits." Wrong twice over. First: the cache persists page *text* only; the crawler's link discovery consumes *rawHtml* on live fetches but falls back to a markdown-regex path on cache hits (`crawler.py:249-254`) — so a cache-served run discovers a structurally different candidate set than a live run, selects different children, and live-fetches every URL the baseline never cached (the observed avif failures were markdown-only image candidates that the HTML path can't even produce, plus `.avif` missing from `_JUNK_EXTS`). Second: I assumed the baseline itself had been cache-served, making the discovery paths match — one openpyxl query on the baseline's own Acquire Log (`From Cache` column: 345/345 False) would have disproved it in the same turn I gave the advice; the workbook records exactly the provenance needed and I didn't look. The general form: "replay" requires every input of every re-run stage to be pinned (content AND discovery AND scoring), and the baseline's actual execution path to be verified, not assumed. The repo already had the correct comparison pattern (`backend_compare.py` replays a pinned URL list); I should have reached for it. (2026-07-06, failed cache-replay prediction.)

## When swapping an LLM path, audit what the client library already provides before porting hand-rolled infrastructure
Switching the summary-layer design from the Power Automate proxy to the existing Azure path changed more than the endpoint: the proxy's hand-rolled 5xx retry (`llmapi.py`) turned out unnecessary (the OpenAI SDK auto-retries 429/5xx with `max_retries=2` by default — the committed Azure call in `extract.py:325` gets this without any code), and the proxy's hard "no temperature/seed" constraint dissolved (the SDK accepts both; the committed code just never passed them). Two design sections written around a limitation of the OLD transport were revisable to something strictly better — but only because the actual client call site was read rather than assumed equivalent. Separately: what's checkable from committed code (call signature, defaults) vs only from the deployment (seed honoured? quota? keys?) had to be split explicitly — the doc lists the latter as owner-verifiable checks instead of asserting them. (2026-07-06, Azure revision of the summary-layer design.)

## An "exclude X from Y" policy is rarely one filter — enumerate every path X can leak back in
Implementing "unverified claims excluded from grouping/digest" took three coordinated changes, and the obvious one alone would have left the policy visibly violated: (1) the input filter in `group.py:_display_values`; (2) `io_excel.py`'s `claim_index`, which anchored citations on the FIRST occurrence of a claim regardless of verified status — so a claim verified on page B but first extracted unverified on page A would still hyperlink to the unverified Provenance row even after the filter; (3) the "flagged for review" half of the decision, which turned out to depend on a dead styling rule. Before declaring an exclusion policy done, trace where the excluded thing's *identity* (IDs, anchors, links, counts) flows, not just where its value does. (2026-07-06, verified-only enforcement.)

## A styling rule can be dead code from a type mismatch — grep for the literal your data actually produces
`_style_sheet`'s orange unverified flag compared cell text against `"FALSE"`, but openpyxl returns Excel booleans as Python bools and `str(False)` is `"False"` — the rule had never fired on any workbook ever produced. It survived because nobody had asserted the flag's presence in a test; the first test that did (this session) required fixing it first. When a display rule keys on a string form of typed data, check what the round-trip actually yields (bool→Excel→openpyxl→str) rather than what the writer intended. (2026-07-06.)

## Feed a summarizer structure you already trust, not raw volume — it makes faithfulness checkable
The raw-claims vs grouped-scaffolding input decision for the LLM summary layer fell out of the eval, not the prompt: with a closed set of `[C####]`-tagged claims grouped into themes, "no invented facts" becomes set-membership on citations and "no dropped critical facts" becomes "top-3 themes each represented" — both mechanical, free, and runnable inline as a write-time gate before any LLM judge is involved. Raw input would have made both checks judgment calls. When designing an LLM stage, pick the input representation that makes the *verification* cheap, then write the prompt around it. (2026-07-06, summary-layer design.)

## A code docstring can assert an invariant the code never enforces — test the claim, not the comment
`src/group.py` says twice ("real verified claims, never synthesized text") that theme labels are always verified, and `brain/decision-log.md` repeats it. Neither is true: `aggregate.py`'s evidence collection and ranking never reads the `verified` flag, so a cluster's medoid (the theme label / hyperlink anchor) can legitimately be an unverified claim. On the real validation workbook this happened twice as an anchor and ten more times as a cited member — including once inside HORIBA's own Diagnostics-type theme, and once quoted verbatim in the Digest sheet's top-3 line (the first thing a consultant reads). A full-population bulk check (resolve every hyperlink/citation, compare against the target row's own status column) caught this in minutes; spot-checking the two named trace subjects would have missed the Agilent/Sartorius instances entirely since they weren't in the assigned HORIBA/FUJIFILM pair. When a design doc asserts "X is always true of the output," write the one script that checks X against every row, not just the rows you were told to trace. (2026-07-05, ADLM traceability audit.)

## A "verify the mojibake" instruction can itself be a false lead — check your own tooling first
Told to check whether "Marlborough" showing `�` was pre-existing corruption, the first three attempts to inspect the string (bare `print`, redirect to file, `cat` the file) all showed `�` for legitimate accented characters (é, ã, ö) that the openpyxl-loaded string held as correct Unicode codepoints (confirmed via `repr()`/`hex(ord(ch))`). The corruption was in this session's Windows-console cp1252 stdout truncating multi-byte UTF-8 before it ever reached the transcript — not in the workbook. Only writing explicit `encoding='utf-8'` output and reading it back with a tool that respects that encoding revealed the truth. When asked to confirm a suspected data bug, verify your own read/print path can even represent the data correctly before trusting what it shows you — especially non-ASCII text on Windows. (2026-07-05, same audit.)

## An overflow "see Provenance" escape hatch can point nowhere once you actually try to use it
`Grouped Themes`' Claim IDs column was truncated to the same 50-item display cap as the bullet list, so a 115-item theme showed 50 IDs plus a bare "(+65 more)" — the exact 65 items the overflow text sends you to Provenance to find had no ID to search by, and Provenance has no Theme column to filter on instead. This is the kind of bug that only shows up when you follow the chain for the pathological large-cell case the brief specifically named (HORIBA), not the small clean one (FUJIFILM) — the small case traced perfectly and would have given false confidence if checked alone. Always trace the largest/ugliest instance of a feature, not just a representative one. (2026-07-05, same audit.)

## A "blocked" status can be three unrelated bugs wearing the same label — open the actual bytes
Extending the bake-off to 25 entities, four showed 0/15 fetch-ok and every row said `robots_disallowed`. Trusting the label would have written off self-hosted crawling on those domains as policy-blocked. Fetching the same robots.txt with `urllib.request` (default UA) vs `curl -A "<honest UA>"` showed 403 vs 200 — `RobotFileParser.read()` never sends the pipeline's own declared User-Agent, only the page-fetch path does, so a WAF that blocks bare `Python-urllib/x.y` (common) makes `disallow_all=True` fire on a robots.txt that actually says `Allow: /`. Meanwhile a fifth entity's "ok" pages were a real Akamai block (`errors.edgesuite.net` reference page in the extracted text), and a sixth's "ok" pages were a real but silently-wrong hydration-timing bug (13 different product URLs, byte-identical 333-char generic shell). Three different root causes, one status column. Always read the actual extracted text/HTML for a sample of each failure/success bucket before writing a taxonomy — the status string is a hypothesis, not a diagnosis. (2026-07-05, 25-entity backend bake-off.)

## An "ok" status can be more dangerous than a failure
The Aladdin Scientific hydration-timing bug didn't fail — it passed the quality gate 14/14 times while returning the same generic storefront chrome for 13 distinct product pages, because the gate only checks length/link-density/retention-ratio, not whether the content matches the requested URL. A downstream cell-population count would show these as fully successful fetches with plausible-looking short answers. When a backend's failure mode can be "confidently wrong" rather than "visibly absent," pass-rate alone cannot catch it — sample and read the actual text on repeated-length or repeated-content signatures even among the "ok" rows, not just the failures. (2026-07-05, same bake-off.)

## A known, logged bug from a prior session can still be unfixed weeks later — check the code, not the log entry
The 2026-07-02 bake-off diagnosed a consent-wall/hydration-timing failure on Bruker and called it "fixable, not a backend limitation" — but no fix was made (settle_ms is still a flat 2000ms in both `playwright_pool.py` and `fetcher.py`, confirmed by reading both files this session), so the exact same failure signature reappeared on a different site (Aladdin) three sessions later. A decision-log entry that says "diagnosed as fixable" is not evidence the fix landed; grep the current code for the specific constant before assuming a documented finding was acted on. (2026-07-05.)

## Missing capability can have two independent causes that both need stating
This machine can't run the Extract layer — not just because the internal LLMAPI/VPN path is unreachable (as predicted), but *also* because there is no `.env` file at all, so even the public Azure endpoint (which IS network-reachable, confirmed via a bare GET) has no API key. Reporting only "no VPN" would imply a laptop with VPN access is sufficient; it also needs the credentials file. When a capability gap has more than one independent blocker, name all of them — fixing one without the other still doesn't unblock the task. (2026-07-05.)

## Build a null-check into every measurement harness
The filter recalibration re-scored cached pages with the OLD queries alongside the new ones; the old-query AUC reproduced the baseline to three decimals, proving the harness itself was faithful before we trusted the improvement (0.636→0.728). A before/after comparison without that null leg can't distinguish "the fix worked" from "the harness measures differently." (2026-07-03, filter validation.)

## Absolute cosine thresholds don't transfer across embedding anisotropy — calibrate in the deployed geometry
Grouping worked perfectly on orthogonal test vectors and produced one giant blob on real claims at every threshold: claims within a cell share a dominant company/domain component that compresses raw cosines into a narrow band (the same compression seen in filter scores). Fix was per-cell mean-centering, not threshold tuning. Rule: whenever a similarity threshold "works in tests but not on real data," suspect the geometry before the threshold — and calibrate on real data from the deployed domain. (2026-07-03, group calibration.)

## Optimise the metric the error asymmetry demands, not the default one
The threshold sweep's best-F1 operating point for R&D location had recall 0.526 — F1 balances precision and recall, but a filter false negative is an unrecoverable lost answer while a false positive costs one LLM call. The right operating point is recall-first (≥0.95), chosen from the same sweep. Defaults like best-F1 encode an error trade-off; check it matches yours. (2026-07-03, filter thresholds.)

## Audit subagent reports with cheap mechanical checks, not re-reads
The grouping agent reported "11 new tests"; `pytest --collect-only` showed 12 — harmless here, but the same class of error could hide a failure. Counts, SHAs, and table numbers from a subagent are one `--collect-only` / one script re-run away from verified; always spend that minute. Same session: re-ran both diagnostics from the merged main tree rather than trusting the worktree output. (2026-07-03, filter/grouping integration.)

## Parallel worktree agents work well when briefs are prescriptive and file-disjoint
Two agents built the filter fix and the grouping layer simultaneously in isolated worktrees; the only shared file (config.py) auto-merged because each added a distinct section. The briefs specified exact function signatures, cache-key pitfalls, and test cases — the agents' judgment went into implementation, not scope. Confirmed approach for independent multi-file changes. (2026-07-03.)

## Check the execution environment before promising a measurement
The task said "re-measure AUC using the cached validation data" — a 30-second probe showed this machine has neither Ollama nor the ADLM cache (both live on the work laptop). Discovering that BEFORE spawning implementation agents reshaped the deliverable honestly: build + unit-test + before-AUC here, one-command after-AUC script for the laptop, stated as pending rather than asserted. (2026-07-03, filter/grouping build.)

## Match the metric to the question type, or it will lie to you
Cell-level recall ("cell has ≥1 answer") said a 6-page crawl cap kept 92% of quality; item-level recall (distinct claims kept) showed it lost 50% of Diagnostics-type items. The first metric nearly shipped a bad recommendation; the honest metric reversed it. For list questions, always count items, not cells. (2026-07-02→03, crawl-budget analysis.)

## Claim/release discipline: every reserved resource needs a release on every non-completion path
The locale-dedup key was claimed at queue time but never released on fetch failure or threshold re-check skip — permanently blackholing all sibling variants of a page that was never actually acquired. Pattern to check whenever something is "claimed early": enumerate every path where the claimed work doesn't complete, and release on each. (2026-07-03 code review finding #1.)

## Measure before designing — existing logs often already contain the diagnosis
"The filter doesn't work" became "the filter embeds the 3-word column name and discards the 30-word instruction (filter.py:101), AUC 0.64" using only the Filter Log + Provenance from a run that already existed. No new runs, no credits, and the fix became obvious and testable. (2026-07-03, filter diagnosis.)

## Bake-off before trusting a backend — and read the failure signature, not just the rate
playwright_pooled looked complete in code review but failed 57% of pages in measurement. The tell was qualitative: identical 2,539-char extractions on every Bruker page = consent-wall text captured before hydration. A pass-rate alone wouldn't have located the cause; the repeated-constant signature did. (2026-07-02 backend bake-off.)

## Sub-agent findings are candidates, not facts — verify against the code before asserting
The review finders claimed the plant-milk workbook has duplicate-domain rows and that a case-mismatch bug had high impact; direct reading showed the first was unverifiable from this machine and the second low-hit-rate. Report verified facts as facts, the rest as flagged risk. (2026-07-03 code review verification pass.)

## Machine-local mode choices belong in .env, not uncommitted config edits
The work laptop's FILTER_MODE=passthrough edit collided with git pull twice before being moved to an env override. Any per-machine setting that lives as a working-tree edit is a recurring merge conflict with a deadline attached. (2026-07-02→03.)

## Honest capability boundaries beat optimistic promises
Free proxies and stealth anti-bot were requested and declined with reasons (security hazard; indefensible posture); the alternative offered was a measured bake-off with pre-registered thresholds. The bake-off then failed, which is the system working — the refusal + measurement protected the deliverable. (2026-07-02.)
